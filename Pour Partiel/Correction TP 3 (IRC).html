Exercice 2 - Suite de Syracuse

Question 1

Il faut bien faire attention à chaque terme de l’énoncé : on demandait ici une fonction qui renvoie (càd retourne) la suite de Syracuse sous forme de liste.

Le principe est relativement simple : à chaque appel de la fonction syracuse, on crée une liste l constituée au départ uniquement de la valeur n courante

  • si n == 1, on retourne directement l
  • sinon, on retourne la liste résultant de la concaténation de l avec la liste obtenue récursivement pour les valeurs suivantes de la liste.
def syracuse(n):
    l = [n]
    if n == 1:
        return l
    elif n % 2 == 0:
        return l + syracuse(n // 2)
    else:
        return l + syracuse(3 * n + 1)

💡 Il faut avoir conscience ici qu’une nouvelle liste (donc un nouvel objet) est instanciée à chaque appel récursif. Chacune de ces liste n’est constituée que d’un entier, donc l’instanciation est très rapide, mais il faut garder à l’esprit que l’instanciation d’un objet demande un certain nombre d’opérations à l’interpréteur / au compilateur ; dans des cas où les objets à instancier sont beaucoup plus complexes, l’instanciation de nombreux objets peut être coûteuse à la fois en temps et en espace. La solution alternative suivante a l’avantage de n’instancier qu’un seul objet l une fois pour toutes :

def syracuse(n, l):
  l.append(n)
  
  if n==1:
    return l
  elif n % 2 == 0:
    return syracuse(n//2, l)
  else:
    return syracuse(3*n+1, l)
-------------------------------------------------
l = []
print(syracuse(28, l))

💡 A chaque appel à la fonction syracuse, on ajoute dans la même liste l la valeur courante de n. Comme on a besoin de cette même liste quand on rappelle récursivement la fonction, on n’a pas d’autre choix que de la passer en paramètre (ou d’en faire une variable globale, mais on rappelle que l’utilisation de variables globales constitue dans la très grande majorité des cas un défaut de conception).

Cette fonction a cependant elle aussi un défaut : elle oblige l’utilisateur à spécifier une liste (vide) lors du premier appel à syracuse. C’est une mauvaise pratique : un utilisateur ne devrait avoir à passer en paramètre que ce dont il a réellement besoin ; or ici, on veut juste calculer la suite à partir d’une valeur n donnée. Le fait que l’algorithme a besoin de gérer une liste l à chaque étape devrait être transparent à l’utilisateur. Pour cette raison, on crée généralement une fonction portant le nom et les paramètres attendus par l’utilisateur, et tout l’algorithme est spécifié dans une fonction qu’on appelle souvent Helper. Ce qui nous donne ici :

def syracuse(n):
  return syracuseHelper(n, [])


def syracuseHelper(n, l):

  if n==1:
    return l
  elif n % 2 == 0:
    return syracuseHelper(n//2, l)
  else:
    return syracuseHelper(3*n+1, l)

Question 5

Dans cette question, il faut mettre en œuvre la mémoïsation à l’aide d’une liste. Il nous faut donc un moyen de déterminer si le temps de vol d’un entier kkk a déjà été calculé et figure dans la liste ; le plus simple est donc d’initialiser la liste des NNN temps de vols avec une valeur “impossible” (par exemple 0, car un temps de vol ne peut être nul, ou, peut-être mieux, avec la valeur None). Il suffit alors de regarder si la kkk-ième case contient une valeur valide.

⚠️ Il faut prêter garde au fait que, dans les calculs, le nombre dont on cherche le temps de vol peut être bien supérieure à NNN ! Par exemple, l’altitude maximale de la suite du nombre 266232662326623 est 106358020106358020106358020 ; autrement dit, on a besoin de connaître le temps de vol du nombre 106358020106358020106358020. Il nous faudrait donc une liste d’au moins 106358020106358020106358020 éléments… Aussi, pour simplifier, on n’utilise la mémoïsation que si le nombre courant est inférieur à NNN.

# Question 2.5 : mémoïsation à l'aide d'une liste

import time

def syracuse2(n):
    if n == 1:
        return 1
    elif n % 2 == 0:
        if n // 2 < N and tps_vol[n // 2]:
            return tps_vol[n // 2] + 1
        return syracuse2(n // 2) + 1
    else:
        if 3*n+1 < N and tps_vol[3*n+1]:
            return tps_vol[3*n+1] + 1
        return syracuse2(3 * n + 1) + 1


# Programme principal
N = 10**6
tps_vol = [None] * (N+1)

start = time.time()
for k in range(1, N+1):
    tps_vol[k] = syracuse2(k)
end = time.time()
print("Temps de calcul : " + str(end - start))

Le temps de calcul a été considérablement amélioré !

💡 Dans ce programme, les temps de vol sont enregistrés dans la liste tps_vol au fur et à mesure des itérations sur la variable k : on stocke d’abord tps_vol[1], puis tps_vol[2], puis tps_vol[3], etc. Pourtant, dans le calcul de tps_vol[3], on est amené à calculer tps_vol[10], tps_vol[5], tps_vol[16]… qu’on ne mémorise pas. C’est dommage, puisqu’on devra les recalculer par la suite ! La version suivante est donc une légère amélioration du programme précédent, qui mémorise un temps de vol dès qu’il a été calculé (il faut bien veiller à ne pas sortir des limites de la liste) :

Question 5 : version avec récursion terminale

Par définition de la suite de Fibonacci, pour calculer un terme on n’a besoin que des deux précédents :
F2=1+1F3=2+1F4=3+2F5=5+3F6=8+5\begin{aligned} F_2&=1+1\\ F_3&=2+1\\ F_4&=3+2\\ F_5&=5+3\\ F_6&=8+5\\ \end{aligned}F2​F3​F4​F5​F6​​=1+1=2+1=3+2=5+3=8+5​

On remarque que sur chaque ligne :

  • le premier terme de l’addition est la somme des termes de la ligne précédente
  • le second terme de l’addition est le premier terme de la ligne précédente

Dit autrement, si Fk=a+bF_k = a + bFk​=a+b, alors Fk+1=(a+b)+aF_{k+1} = (a + b) + aFk+1​=(a+b)+a. Il suffit donc, à chaque itération, de transmettre à l’itération suivante les valeurs a+ba+ba+b et aaa.

Dans une première version, on peut utiliser une variable i pour se rappeler le numéro de l’itération en cours ; quand i atteint la valeur n demandée, on renvoie le résultat :

def fibonacci3(n):
  return fiboHelper(0, n, 1, 1)

def fiboHelper(i, n, a, b):
  if i == n:
    return b
  else:
    return fiboHelper(i+1, n, a + b, a)

💡 N’hésitez pas à exécuter ce code avec pythontutor pour bien le comprendre !

💡 La fonction principale fibonacci3 pourrait afficher le résultat plutôt que le retourner. Cependant, il faut toujours se mettre à la place d’un utilisateur de votre code, qui pourrait avoir besoin de vos fonctions dans ses propres programmes. Avoir une fonction qui retourne une valeur permet de stocker cette valeur dans une variable et d’en faire ce qu’on veut ensuite ; alors que si une fonction affiche quelque chose, on le verra à l’écran mais on ne pourra rien en faire d’autre.

On peut toutefois être encore plus astucieux et se passer de la variable i en comptant tout simplement les itérations à l’envers (on part de n, et on s’arrête quand on a atteint 0) :

  return fiboHelper(n, 1, 1)

  if n == 0:
    return b
  else:
    return fiboHelper(n - 1, a + b, a)

Exercice 4 - Diviser pour régner : maxliste

On découpe la liste en deux sous-listes dont on calcule récursivement le maximum (le cas de base étant le cas d’une liste à un seul élément, pour laquelle on obtient immédiatement le maximum). Il suffit alors de renvoyer le plus grand élément entre le maximum de gauche et le maximum de droite.

def maxliste(l):
  if len(l) == 1:
    return l[0]
  else:
    maxG = maxliste(l[:len(l)//2])	# demi-liste de gauche
    maxD = maxliste(l[len(l)//2:])  # demi-liste de droite
    return maxG if maxG > maxD else maxD

Exercice 5 - Tours de Hanoï

Le principe de l’algorithme récursif permettant de résoudre le problème des tours de Hanoï, qui consiste à déplacer n disques d’un pilier source vers un pilier destination en utilisant un pilier auxiliaire, est extrêmement simple… une fois qu’on l’a compris.
Comme on l’a vu en cours, il faut procéder comme pour résoudre une récurrence en maths.

Rappel : par exemple, pour démontrer par récurrence que 1+2+...+n=n(n+1)21+2+...+n=\frac{n(n+1)}{2}1+2+...+n=2n(n+1)​ :

  1. on vérifie que cette formule est vraie pour le cas de base : ici, la formule est vraie pour n=1n=1n=1 : 1=1×221=\frac{1\times 2}{2}1=21×2​
  2. on suppose que la formule est vraie à l’ordre n−1n-1n−1
  3. on la démontre à l’ordre nnn :
    1+2+...+n=(1+2+...+n−1)+n=(n−1)n2+n (par hypotheˋse de reˊcurrence)=(n−1)n2+2n2=(n−1+2)n2=n(n+1)2\begin{aligned} 1+2+...+n &= (1+2+...+ n-1) + n\\ &=\frac{(n-1)n}{2} +n \text{ (par hypothèse de récurrence)}\\ &=\frac{(n-1)n}{2} +\frac{2n}{2}\\ &=\frac{(n-1+2)n}{2}=\frac{n(n+1)}{2} \end{aligned}1+2+...+n​=(1+2+...+n−1)+n=2(n−1)n​+n (par hypotheˋse de reˊcurrence)=2(n−1)n​+22n​=2(n−1+2)n​=2n(n+1)​​
    Donc la formule est vraie pour tout nnn.

Ainsi, pour résoudre le problème des tours de Hanoï, il faut :

  1. montrer qu’on peut résoudre le problème dans le cas de base : ici, il s’agit du cas où on doit déplacer n=1 disque ; sa résolution est immédiate : on prend le disque du pilier source et on le déplace vers le pilier destination ;

  2. supposer qu’on sait résoudre le problème à n-1 disques : autrement dit, on suppose qu’on dispose d’un algorithme / d’une boîte noire capable de déplacer correctement n-1 disques d’un pilier vers un autre en utilisant le troisième pilier comme auxiliaire

  3. utiliser cette hypothèse / boîte noire pour résoudre le problème à n disques :
    pour déplacer n disques du pilier source vers le pilier destination en utilisant le pilier auxiliaire, il faut :

    1. déplacer n-1 disques du pilier source vers le pilier auxiliaire en utilisant le pilier destination (ce qu’on sait faire par hypothèse, et parce que le pilier destination est bien libre) ;
    2. déplacer 1 disque du pilier source directement vers le pilier destination (c’est possible puisqu’on ne déplace qu’un seul disque) ;
    3. déplacer les n-1 disques du pilier auxiliaire vers le pilier destination en utilisant le pilier source (ce qu’on sait faire par hypothèse, et parce que le pilier source est bien libre).

On obtient ainsi immédiatement l’algorithme récursif :

def hanoi(n):
   hanoiHelper(n, 1, 3, 2)

def hanoiHelper(n, s, d, a):   # nb disques / source / destination / auxiliaire
  if n == 1:
    print(f"Déplacer un disque de {s} vers {d}")
  else:
    hanoiHelper(n-1, s, a, d)
    hanoiHelper(1, s, d, a)
    hanoiHelper(n-1, a, d, s)

Exercice 6 - Recherche dichotomique

Question 1 : recherche séquentielle

  • Le meilleur des cas se produit quand l’élément cherché est le premier de la liste ; une seule comparaison est alors effectuée, et l’algorithme est en O(1)O(1)O(1) (temps constant) ;
  • le pire des cas se produit quand l’élément cherché est le dernier de la liste ; il faut alors comparer l’élément cherché à tous ceux de la liste : l’algorithme est en O(n)O(n)O(n) ;
  • en moyenne, on effectuera 1/n×(1+2+3+...+n)1/n \times (1 + 2 + 3 + ... + n)1/n×(1+2+3+...+n) opérations, soit (n+1)/2(n+1)/2(n+1)/2 : en moyenne, l’algorithme de recherche séquentielle effectue au plus O(n)O(n)O(n) opérations.

Question 2 : recherche dichotomique

On considère cette fois qu’on recherche un nombre k dans une liste l triée.
Principe : on commence par regarder la valeur l[taille/2] au milieu de la liste :

  • si k=l[taille/2] : on a trouvé le nombre k, on peut retourner sa position ;
  • si k<l[taille/2] : inutile de considérer la partie droite de la liste (parce qu’elle est triée), on se concentre sur la partie gauche ;
  • si k>l[taille/2] : inutile de considérer la partie gauche de la liste (parce qu’elle est triée), on se concentre sur la partie droite.

Exemple : on cherche le nombre 57 :

Commençons par écrire une fonction qui renvoie True ou False selon que le nombre k se trouve dans la liste l ou non :

def dichoV1(l, k):
  if len(l) == 0: # si on finit par tomber sur une liste à 0 éléments, c'est que k n'est pas présent
      return False
  
  m = len(l)//2  # <- milieu de la liste
  if l[m] == k:
    return True
  elif k < l[m]:
    return dichoV1(l[:m], k)   #l[:m] : moitié gauche de la liste
  else:
    return dichoV1(l[m+1:], k) #l[m:] : moitié droite de la liste

L’énoncé comportait cependant une petite difficulté supplémentaire : on ne voulait pas seulement savoir si k est dans la liste ou non, mais on voulait aussi sa place dans la liste s’il est présent, et -1 sinon. Avec cette contrainte, on ne peut travailler récursivement sur une sous-liste de la liste précédente, car on perd toute notion d’indice !

Exemple : sur l’exemple précédent, à la première itération, on a liste de 10 éléments indexés de 0 à 9 ; dans cette liste, 28 est à l’indice 5 et 71 à l’indice 9. Mais à l’itération suivante de l’algorithme, on ne travaille que sur la moitié droite de la liste, et 28 est maintenant à l’indice 0 et 71 à l’indice 4 !

On est donc obligé de conserver à chaque étape la liste initiale. Pour savoir quelle sous-partie de la liste initiale on est en train de traiter, on utilise deux marqueurs get d (pour gauche et droite) ; si g finit par se retrouver à droite de d, c’est que le nombre k n’est pas dans la liste.

Au final, une solution possible est la suivante :

 # Paramètres :
 # l : liste de nombres *triée*
 # k : nombre cherché
def dicho(l, k):
  return dichoHelper(l, k, 0, len(l) -1)

# Paramètres :
# l : liste de nombres
# k : nombre cherché
# g : position du marqueur gauche
# d : position du marqueur droite
def dichoHelper(l, k, g, d):
  m = (g + d) // 2  # <- milieu du tableau

  if g <= d:  
      if l[m] == k:
        return m
      elif l[m] > k:
        return dichoHelper(l, k, g, m -1)
      else:
        return dichoHelper(l, k, m+1, d)

  else:
    return -1

Solution via le Master Theorem

Pour rappel, le Master Theorem s’applique aux relations du type T(n)=aT(n/b)+ncT(n) = aT(n/b) + n^cT(n)=aT(n/b)+nc (avec a≥1a \geq 1a≥1, b>1b>1b>1 et c≥0c \geq 0c≥0). Ici :

  • a=1a = 1a=1
  • b=2b = 2b=2
  • c=0c = 0c=0 (une constante est un polynôme de degré 0).

On doit ensuite comparer ccc et log⁡ba\log_b alogb​a ; les deux valeurs sont identiques, on tombe donc dans le deuxième cas du Master Theorem, qui nous dit immédiatement que T(n)=O(nc⋅log⁡n)=O(log⁡n)T(n) = O(n^c \cdot \log n) = O(\log n)T(n)=O(nc⋅logn)=O(logn).
